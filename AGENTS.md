# 行动规范文档 (Agents Guidelines)

## 0. 接手导航（新 Agent 必读）
- **规格说明**：以根目录 `spec.md` 为当前版本唯一规格来源，先理解规则后再改动实现。撰写新版规格说明时请注意：`spec.md`应该整个产品特性的完整体现，而非当前版本的增量变更。
- **任务目标来源**：以根目录 `tasks.md` 作为研发任务拆解与执行顺序依据。
- **进度与完成定义**：以根目录 `checklist.md` 作为验收标准，任务完成需能映射到验收项。撰写checklist时应该注意，checklist是当前版本的全部特性的验收清单，涵盖所有产品特性的检查点，对于已完成的特性，只要没有变更，应该保留检查点（好比冒烟测试）。对于变更的特性应该采用最新说法。
- **历史归档入口**：`docs/` 仅用于历史对照，不作为当前实现约束优先级。

## 1. 架构与职责分离
- **纯逻辑层 (Game Logic)**：网格状态、积木生成、放置校验、消除判定、计分计算必须提取为纯函数或独立的自定义 Hook (如 `useGameLogic`)。确保核心算法独立于 UI 组件，便于单元测试和维护。
- **UI 展示层 (View)**：负责渲染网格、备选区和分数面板。UI 状态应完全由逻辑层驱动（数据驱动视图），UI 组件应尽量保持无状态（Stateless）。

## 2. 状态管理规范
- **不可变性 (Immutability)**：更新网格状态时，必须返回全新的二维数组引用，严禁直接修改原数组（Mutate）。
- **单一数据源**：当前得分、网格状态、备选区积木状态应集中在顶层管理，避免组件间状态同步导致的不一致。

## 3. 交互与体验规范
- **拖拽与预览**：拖拽积木进入网格悬停时，需提供高亮预览（Ghost / Preview），明确提示玩家松手后积木的落点及该位置是否合法。
- **动画过渡**：
  - 消除行/列时，需有短暂的消除动画（如方块闪烁或缩小消失），提升爽快感。
  - 新积木生成时有淡入或滑动出现的视觉反馈。
- **响应式设计**：确保游戏界面在移动端（竖屏）和 PC 端（横屏）均有良好的显示效果，同时兼容鼠标点击/拖拽与触摸屏交互。

## 4. 错误处理与边界条件
- 严密校验放置越界、重叠。
- Game Over 判定必须在“每次放置成功后”和“每次生成新积木后”精确执行。算法需遍历场地所有空格与备选区剩余积木进行匹配测试。
